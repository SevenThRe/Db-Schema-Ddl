<task type="auto">
  <name>Task 1: Expand the shared runtime contract for page-first execution, export, and schema context</name>
  <files>shared/schema.ts, src-tauri/src/db_connector/mod.rs</files>
  <read_first>
    - shared/schema.ts (current DbConnectionConfig, QueryExecutionRequest, DbQueryBatchResult, FetchMoreRequest, ExportRowsRequest, DbSchemaSnapshot, BinaryCommandResult)
    - src-tauri/src/db_connector/mod.rs (Rust-side serde structs that must match shared/schema.ts)
  </read_first>
  <action>
In both `shared/schema.ts` and `src-tauri/src/db_connector/mod.rs`, make the runtime contract explicit instead of inferring behavior from `rows.length` and `totalRows`.

1. Keep `DbConnectionConfig.defaultSchema`, but treat it as an active runtime concept by ensuring all query/introspect/explain/export request structs can carry an optional `schema`.

2. Replace the old query-batch assumptions with explicit paging metadata. Add these fields to `DbQueryBatchResult` in both TS and Rust:
   - `returnedRows: number`
   - `hasMore: boolean`
   - `pagingMode: "offset" | "none" | "unsupported"`
   - `pagingReason?: string`
   - `nextOffset?: number`
   - `schema?: string`
   - change `totalRows` to optional nullable form so the first page path does not require a full count before returning data

3. Update `QueryExecutionRequest`, `FetchMoreRequest`, and `ExplainRequest` to carry:
   - `schema?: string`
   - `limit` and `offset` semantics that are for runtime paging, not post-fetch trimming

4. Replace the old client-only `ExportRowsRequest` with a runtime-backed export contract:
   - request fields: `connectionId`, `requestId`, `sql`, `schema?`, `format`, `scope`, `batchIndex?`, `loadedRows?`, `columns?`, `maxRows?`
   - `scope` must be one of `current_page`, `loaded_rows`, `full_result`
   - response type must be `BinaryCommandResult` so the desktop bridge can download the backend-generated artifact

5. Extend `DbSchemaSnapshot` with `schema: string` so the UI can display which PostgreSQL schema the object tree and introspection data came from.

6. Add a schema-list response type that the UI can consume directly. A plain `string[]` is acceptable, but it must be reflected in TS and Rust signatures as the canonical contract.
  </action>
