# Phase 17: Safe Data Editing - Context

**Gathered:** 2026-04-08
**Mode:** Smart Discuss (autonomous defaults, non-interactive)
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement the safe single-table row editing loop inside the existing DB Workbench path so DATA-01/02/03 become true:
- explicit edit eligibility (single-table + provable primary-key mapping + non-readonly connection)
- SQL preview and affected-row summary before commit
- all pending edits committed in one transaction with full rollback on any failure

Out of scope in this phase: multi-table arbitrary SQL editing, broad sync workflows, and cross-table mutation tooling.

</domain>

<decisions>
## Implementation Decisions

### 1) Edit eligibility gate (DATA-01)
- Edit mode is enabled only for result batches that come from the object-explorer table path (`Open table` / `Select top 100` / `Select explicit columns`) and resolve to exactly one table.
- `Count rows` result sets are never editable.
- Eligibility is fail-closed and requires all of the following:
  - connection is not readonly
  - result batch has no execution error and has at least one row
  - source table exists in the active schema snapshot
  - every primary-key column for that table is present in returned result columns
  - each loaded row yields a unique primary-key tuple
- If any check fails, grid remains read-only and UI shows the blocking reason.

### 2) Patch model and SQL preview contract (DATA-02)
- Grid editing uses a client-side patch set keyed by `(rowPrimaryKeyTuple, columnName)`.
- Primary-key columns are locked (not editable) in phase 17.
- No-op edits (new value equals original value) are removed from the patch set automatically.
- SQL preview is generated by Rust, not by frontend string concatenation.
- Introduce a prepare step (`db_grid_prepare_commit`) that returns:
  - normalized affected-row count
  - changed-column summary
  - SQL preview lines (capped, with truncation flag)
  - `planId` + `planHash` for commit integrity

### 3) Transaction commit + rollback semantics (DATA-03)
- Commit path is two-step: `prepare` then `commit`.
- `db_grid_commit` executes the prepared patch plan in one DB transaction on one connection.
- Any statement failure triggers full rollback; partial success is not allowed.
- On successful commit, frontend clears pending patches and re-runs the originating table query to refresh data.
- User-side cancel/discard before commit only clears local patches and does not call backend mutation commands.

### 4) Capability and wiring defaults
- Add explicit `db.data.edit` capability and gate edit APIs on it.
- Add new host API + desktop bridge methods for prepare/commit flow; keep current query/explain/export APIs unchanged.
- Add dedicated Rust module (`grid_edit.rs`) and command registration in `src-tauri/src/lib.rs`; avoid expanding `commands.rs` with mutation transaction logic.
- Phase 17 mutation scope is `UPDATE` only; INSERT/DELETE are deferred.

### Recommended defaults for planner
- Prefer deterministic safety over permissive editing.
- Prefer backend-generated SQL plans over frontend-generated SQL strings.
- Prefer explicit "not eligible" reasons in UI over silent disable states.

</decisions>

<specifics>
## Specific Implementation Notes

- For PostgreSQL, edit operations must use the same active schema resolution model already used by query/export (`request schema -> defaultSchema -> public`).
- Identifier handling should follow existing safe quoting conventions; values must always use parameter binding.
- SQL preview should show representative statements and aggregate counts, not an unbounded dump for large edit sets.
- Failure responses should include enough row-key context for the UI to point users to the failed edit group.
- Regression tests should cover:
  - eligibility true/false matrix (readonly, missing PK, duplicate PK tuple, unsupported source)
  - prepare/commit handshake integrity (`planHash` mismatch rejection)
  - transaction rollback guarantees on mid-batch failure

</specifics>

<deferred>
## Deferred Ideas

- Editing arbitrary SQL result sets (JOIN/CTE/multi-table projections).
- INSERT/DELETE row operations and bulk row append workflows.
- Optimistic locking/version-column conflict handling.
- Commit-time cancellation and resumable mutation batches.
- Bridging grid edit plans into phase-18 live sync apply pipelines.

</deferred>
