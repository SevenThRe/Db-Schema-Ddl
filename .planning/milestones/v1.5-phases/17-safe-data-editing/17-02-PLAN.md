  <action>
Implement atomic commit semantics for prepared grid edits.

1. In `db_grid_commit`:
   - load prepared plan by `planId` from registry.
   - reject when `request.plan_hash != prepared.plan_hash`.
   - open one DB transaction on the same connection/pool used by prepare.
   - execute prepared UPDATE statements in deterministic order.
   - on first statement error:
     - rollback transaction,
     - return failure with row-key context and failing SQL index.
   - on success:
     - commit transaction,
     - remove consumed plan from registry,
     - return committed row count.
2. Ensure commit endpoint cannot execute arbitrary SQL payloads from frontend; only server-prepared statement templates may run.
3. Preserve Phase-17 scope by rejecting INSERT/DELETE mutations in commit path.
