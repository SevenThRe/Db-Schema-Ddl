# v1.1 Milestone Proposal

**Prepared:** 2026-03-18
**Granularity:** Coarse
**Status:** Draft for main-thread review and integration

## Proposal Summary

`v1.1` should extend the current DB management release in two directions:

1. Direct `DB vs DB` comparison inside `DB 管理`, including directional diff review and preview.
2. Reverse authoring workflows that provide first-party `.xlsx` templates first, then turn supported DDL into parser-compatible `.xlsx` files.

This proposal keeps the current `v1.0` planning artifacts intact. The intent is for the main thread to review this draft, then open a new `v1.1` milestone and integrate the accepted content without overwriting the audit trail of the completed `v1.0` milestone.

## Milestone Goal

Users can compare two real database environments directly and can move between SQL definitions and the product's Excel-based schema documents without hand-building workbook structure.

## Scope Guidance

### In Scope

- Live `DB vs DB` compare between two saved connection/schema targets
- Directional preview of how a target differs from a selected source DB
- Supported `DDL -> canonical schema -> XLSX` workflow
- Built-in `.xlsx` template workflows aligned to the existing parser
- Round-trip validation so generated/template workbooks reopen cleanly

### Out of Scope for v1.1

- Automatic `DB -> DB` apply or sync execution
- General SQL editor or arbitrary SQL parsing
- Support for every dialect-specific DDL feature on day one
- Custom user-designed Excel layouts outside the supported template family
- Non-table object coverage beyond the current canonical scope unless explicitly added later

## Proposed v1.1 Requirements

### DB-vs-DB Compare

- [ ] **DBDB-01**: User can choose a source DB and a target DB from saved connections and compare their schemas directly
- [ ] **DBDB-02**: User can inspect `DB vs DB` differences in the `DB 管理` workspace with clear source/target labels, filters, and object drilldown
- [ ] **DBDB-03**: User can review ambiguous rename or equivalence candidates before directional preview proceeds
- [ ] **DBDB-04**: User can generate a non-applying directional preview that shows how the target DB would need to change to match the selected source DB

### XLSX Templates and Round-Trip Validation

- [ ] **TPL-01**: User can create a new schema-definition workbook from a built-in `.xlsx` template aligned to the supported Japanese header layout
- [ ] **TPL-02**: User can choose a template variant for `multi-table per sheet` versus `table per sheet` authoring
- [ ] **TPL-03**: App can round-trip validate generated or templated `.xlsx` outputs by reopening them through the parser and surfacing mismatches before the user trusts them

### DDL-to-XLSX Conversion

- [ ] **DDLX-01**: User can paste or import supported MySQL `CREATE TABLE` DDL and parse it into the app's canonical schema model
- [ ] **DDLX-02**: App surfaces unsupported or lossy DDL constructs before workbook export so the user knows what requires manual cleanup
- [ ] **DDLX-03**: User can export parsed DDL into an `.xlsx` workbook that the existing Excel parser can open without format surgery
- [ ] **DDLX-04**: Oracle DDL import remains explicitly deferred until the MySQL-first reverse-authoring workflow is stable

## Phase Details

### Phase 1: Cross-Database Compare
**Goal**: Users can compare one live DB environment against another inside `DB 管理` and review a directional preview without applying changes.
**Depends on**: v1.0 Phases 3, 4, and 5
**Requirements**: DBDB-01, DBDB-02, DBDB-03, DBDB-04
**Success Criteria**:
1. User can choose two saved connection/schema targets and run a direct compare without exporting an intermediate `.xlsx` or snapshot file manually.
2. User can inspect added, removed, modified, and rename-candidate differences in the existing DB diff workspace with unambiguous source and target context.
3. Ambiguous rename or equivalence cases block directional preview until the user resolves them.
4. User can generate a non-applying preview that explains what would need to change in the target DB to converge toward the source DB.
**Plans**: TBD

### Phase 2: Template and Round-Trip Authoring
**Goal**: Users can start from supported `.xlsx` templates and trust generated workbooks because the app validates that they reopen correctly.
**Depends on**: Nothing beyond the shipped Excel parser and workbook export foundation
**Requirements**: TPL-01, TPL-02, TPL-03
**Success Criteria**:
1. User can create a new workbook from first-party templates that already match the product's supported Japanese header and sheet-layout conventions.
2. User can choose an authoring layout variant that matches either `multi-table per sheet` or `table per sheet` maintenance style.
3. Generated or templated workbooks can be re-imported automatically and either pass round-trip validation or show actionable mismatch warnings before use.
**Plans**: TBD

### Phase 3: DDL Import and XLSX Export
**Goal**: Users can turn supported SQL table definitions into the product's Excel-based document format with explicit fidelity reporting.
**Depends on**: Phase 2
**Requirements**: DDLX-01, DDLX-02, DDLX-03, DDLX-04
**Success Criteria**:
1. User can paste or load supported MySQL table DDL and see the parsed tables in a reviewable canonical form before export.
2. Unsupported or lossy syntax is surfaced explicitly instead of being silently discarded during conversion.
3. User can export the reviewed result into an `.xlsx` workbook that opens through the normal parser flow and preserves the expected tables and columns.
4. Oracle DDL import is kept explicitly out of the first reverse-authoring cut unless later added as a separate requirement expansion.
**Plans**: TBD

## Why These Phases

- Phase 1 stands on its own because `DB vs DB` compare is a direct extension of the shipped DB-management workspace and uses existing canonical-schema, diff, blocker, and preview seams.
- Phase 2 comes before `DDL -> XLSX` because your nearer-term authoring need is first-party templates and trustworthy round-trip validation, not full reverse parsing on day one.
- Phase 3 depends on Phase 2 because `DDL -> XLSX` is much safer once there is already a validated template family and workbook round-trip contract.

## Notes for Planning

- Keep `DB vs DB` strictly compare-and-preview in `v1.1`; do not smuggle live cross-environment apply into the same milestone unless it becomes an explicit requirement.
- Treat `DDL -> XLSX` as a MySQL-first supported subset workflow. The proposal assumes good support for the table/column features already represented by the canonical model, with explicit warnings for anything outside that envelope.
- Fold `v1.0` verification debt into plan-level acceptance where it materially reduces risk, but do not let it consume a dedicated roadmap phase unless the user explicitly reprioritizes around process cleanup.
