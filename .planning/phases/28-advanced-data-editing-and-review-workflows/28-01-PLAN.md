## 28-01 Plan

Objective: expose row delete staging/review/revert/commit in the canonical DB Workbench grid-edit flow.

Implementation scope:

1. Add `pendingDeleteRows` state to `WorkbenchLayout`.
2. Reset delete drafts on connection/schema/result batch changes alongside existing edit reset semantics.
3. Wire delete drafts into `prepareGridCommit`.
4. Surface delete summaries into `GridEditCommitDialog`.
5. Protect delete-staged rows from client-side result-window trimming.
6. Extend `ResultGridPane` with:
   - stage delete / revert delete affordance for selected row
   - pending delete state styling
   - pending delete summary list
   - mixed edit/delete footer counts
7. Keep row-level revert/discard flows coherent when delete drafts exist.
8. Add a small runtime guardrail so committed update/delete statements must affect exactly one row.
9. Add client/server tests covering the new wiring.

Out of scope:

- inserted row drafting
- durable drafts across reruns/restarts
- expanding editable sources beyond current starter/table-open flow
